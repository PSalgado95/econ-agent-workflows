# Project brief template

Use this only when the user approves creating or refreshing a project-backbone document.

Recommended path: `PROJECT_BRIEF.md`, unless the repo already has a clearer convention.

Keep it compact. This is not a dated execution plan, not a README, and not agent instructions.

```md
# <Project or paper title> project brief

## Purpose
- Research question or descriptive object:
- Intended reader:
- Current claim budget:

## Data and sample
- Data vintage:
- Main unit of observation:
- Current sample boundary:
- Important exclusions or missingness rules:

## Definitions and comparisons
- Core definitions:
- Main baseline:
- Benchmark or comparison object:
- Choices that should not drift without approval:

## Current empirical tracks
- Active track:
- Deferred track:
- Known bottleneck:

## Canonical outputs
- Main tables:
- Main figures:
- Review bundle or note surfaces:

## Open decisions
- Decision:
- Why it matters:
- Current default:
```

Minimum quality check:
- it should help a new session understand the paper/data-project state quickly;
- it should not duplicate long plans or logs;
- it should point to dated plans, issues, notes, or outputs rather than copying them wholesale;
- it should not contain raw restricted data or confidential evidence.
